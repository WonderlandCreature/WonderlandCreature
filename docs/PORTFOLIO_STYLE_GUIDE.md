# Enterprise AI Workflow Systems Portfolio Style Guide

## Positioning
This portfolio should present AI, analytics, and workflow projects as business systems that support execution, operational intelligence, and decision-making.

## Core Principle
Do not frame projects as coursework, isolated experiments, or notebook-only analysis.

Frame them as:
- AI-native business systems
- Workflow-driven decision support
- Operational intelligence projects
- Enterprise automation and execution systems

## Language Direction
Avoid:
- homework
- assignment
- class project
- prediction model
- analysis
- dashboard
- ML experiment
- notebook only

Prefer:
- system
- workflow
- decision engine
- operational intelligence
- execution insights
- business system project
- workflow optimization
- architecture

## README Writing Rules
- Start with the business or operational challenge, not the class context.
- Describe the workflow from input to action.
- Explain how the project supports decisions, execution, or efficiency.
- Use architecture, workflow, system design, and operational logic language.
- Keep the tone concise, business-facing, and portfolio-ready.

## Standard Project README Template
```md
# Project Name

## Problem
What business or operational challenge does this system address?

## Objective
What decision, workflow, or operational outcome is this project designed to improve?

## Workflow
Input -> Processing -> Decision -> Action

## System Design
Models, workflow logic, architecture, automation steps, or operational design used.

## Key Insights
Main behaviors, segment patterns, workflow observations, or decision signals discovered.

## Business Impact
How this system improves targeting, execution, visibility, efficiency, or decision-making.

## Future Improvements
Potential next-step extensions, automation layers, or production enhancements.
```

## Repository README Structure
Each repository README should include:
- a short positioning statement
- a `Focus` section
- a `Project Areas` section
- a `Methods and System Patterns` section
- a `Business Relevance` section
- a `Repository Structure` block

## Naming Rules
- Use `kebab-case` for repository names and project directories.
- Prefer names like `coupon-targeting-system` over `coupon_targeting`.
- Prefer business-facing names like `decision-systems`, `workflow-lab`, and `operational-intelligence`.
